# 股票基金数据获取和管理平台：完整落地流程

这个项目的主线始终只有一条：
先做出最小可运行闭环，再扩展，再补文档，再答辩演练。

---

## 第 1 步：先把范围钉死，不允许后面乱加

- 这一步做什么：确认项目边界、明确必做功能和暂缓功能
- 作用和意义：防止后面开发失控，保证你能按时交付

详细操作：
1. 打开课程指导书，只提取题目二的必做点，写成清单。
2. 在 README 里写明项目目标、MVP 功能、非 MVP 功能。
3. 给自己设一个硬规则：必做闭环没通之前，不加任何炫技功能。
4. 新建一个 docs/todo.md，把功能分成三类：必须做、可选做、现在不做。
5. 把答辩要演示的链路提前写出来：登录 -> 触发采集 -> 查询 -> 导出 -> 日志。

完成标准：
- 你能一句话说清楚项目最小闭环是什么。
- 你知道哪些功能绝对不在当前版本做。

预计工作量：4 到 6 小时。

---

## 第 2 步：把工程骨架搭起来，确保能启动

- 这一步做什么：创建前后端目录、初始化依赖、跑通空服务
- 作用和意义：后续所有开发都在这个骨架上进行，先把地基打稳

详细操作：
1. 在项目根目录建立基础结构：
   - backend
   - frontend
   - sql
   - scripts
   - tests
2. 初始化后端环境（Windows PowerShell）：
   - cd backend
   - python -m venv .venv
   - .\\.venv\\Scripts\\Activate.ps1
   - pip install fastapi uvicorn sqlalchemy pymysql alembic pydantic pandas akshare tushare apscheduler python-jose passlib[bcrypt] python-multipart
   - pip freeze > requirements.txt
3. 初始化前端环境：
   - cd ..
   - npm create vite@latest frontend -- --template vue
   - cd frontend
   - npm install
   - npm install element-plus axios pinia vue-router echarts
4. 后端先写一个最小 app，只有 /health 接口。
5. 前端先跑起默认页面，确认 npm run dev 正常。

完成标准：
- 后端服务能启动。
- 前端页面能打开。
- 目录结构稳定，后面不再随意大改。

预计工作量：6 到 8 小时。

---

## 第 3 步：先做数据库，不要先堆业务代码

- 这一步做什么：完成数据库设计和初始化脚本
- 作用和意义：模型稳定后，接口和页面开发才不会反复返工

详细操作：
1. 先确认核心表（建议至少这几张）：
   - users
   - roles
   - instruments
   - stock_daily
   - fund_nav
   - crawl_jobs
   - crawl_runs
   - export_records
2. 在 sql/init.sql 写建表语句、唯一索引、基础索引。
3. 创建数据库并执行脚本：
   - CREATE DATABASE stock_fund_platform DEFAULT CHARSET utf8mb4;
   - USE stock_fund_platform;
   - SOURCE sql/init.sql;
4. 在 backend 下创建 .env，至少配置数据库连接和密钥。
5. 在 docs/db.md 记录表结构、字段含义、索引原因。

完成标准：
- 所有核心表都能成功创建。
- 你能解释每张表的用途和主外键关系。

预计工作量：8 到 10 小时。

---

## 第 4 步：先打通后端最小闭环，不要急着写前端页面

- 这一步做什么：把认证、健康检查、基础路由跑通
- 作用和意义：后端服务稳定后，后面采集和查询才能快速迭代

详细操作：
1. 建立后端分层目录：
   - app/api
   - app/models
   - app/schemas
   - app/services
   - app/core
   - app/tasks
2. 先实现三个最小接口：
   - GET /api/health
   - POST /api/auth/login
   - GET /api/auth/me
3. 做一个最简单 JWT 认证流程（管理员/普通用户两类）。
4. 打开 Swagger 自检，确保请求和响应结构统一。
5. 错误码统一：业务错误、参数错误、权限错误至少区分开。

完成标准：
- 登录可以拿到 token。
- 通过 token 可以访问受保护接口。
- Swagger 文档可直接演示。

预计工作量：8 到 10 小时。

---

## 第 5 步：做采集器，先搞定股票和基金各一条链路

- 这一步做什么：实现股票和基金数据采集、清洗、标准化
- 作用和意义：题目二核心价值就在数据采集和管理，必须尽早落地

详细操作：
1. 在 app/tasks/crawlers 下拆两个采集入口：
   - 股票日线采集
   - 基金净值采集
2. 每次采集都做三类清洗：
   - 缺失值处理
   - 代码标准化
   - 字段类型统一
3. 采集结果写入 stock_daily、fund_nav。
4. 遇到异常时不要中断整个服务，写入 crawl_runs 的失败记录。
5. 给每个采集任务记录来源、开始时间、结束时间、状态、错误信息。

完成标准：
- 手动执行一次采集可以成功入库。
- 失败场景有明确日志。

预计工作量：10 到 12 小时。

---

## 第 6 步：做任务管理，支持手动触发和定时执行

- 这一步做什么：把采集任务做成可管理的任务系统
- 作用和意义：答辩时可以展示系统不是脚本堆砌，而是可管理平台

详细操作：
1. 实现任务相关接口：
   - POST /api/tasks/crawl
   - GET /api/tasks
   - GET /api/tasks/{task_id}
   - GET /api/tasks/{task_id}/logs
2. 使用 APScheduler 增加定时任务机制。
3. 系统启动时加载已启用任务，系统关闭时安全退出调度器。
4. 在任务执行前后写日志，确保过程可追踪。
5. 给任务增加启用/停用开关，避免调试期误触发。

完成标准：
- 你可以从接口手动触发任务。
- 到时间会自动触发任务。
- 能查看最近任务执行历史。

预计工作量：8 到 10 小时。

---

## 第 7 步：做查询和导出接口，让数据“可用”

- 这一步做什么：实现查询分页、筛选、导出
- 作用和意义：没有查询和导出，项目只算采集脚本，不算管理平台

详细操作：
1. 实现查询接口：
   - GET /api/stocks/{code}/daily
   - GET /api/funds/{code}/nav
   - GET /api/instruments
2. 分页参数统一为 page 和 page_size。
3. 支持基础筛选：代码、日期范围、市场或类型。
4. 实现导出接口：
   - POST /api/exports
   - GET /api/exports
5. 导出时落 export_records，记录发起人、参数、结果。

完成标准：
- 查询接口返回稳定。
- 前端后续可直接消费。
- 导出结果可下载，且有历史记录。

预计工作量：8 到 10 小时。

---

## 第 8 步：前端只做 4 个核心页面，先够用再美化

- 这一步做什么：完成最少可答辩页面
- 作用和意义：确保老师能直接看到系统功能闭环

详细操作：
1. 做登录页：用户名、密码、错误提示。
2. 做查询页：
   - 股票查询模块
   - 基金查询模块
   - 表格 + 图表 + 导出按钮
3. 做任务页：
   - 手动触发任务
   - 任务状态列表
   - 日志详情
4. 做仪表盘页：
   - 数据条数
   - 最近任务状态
   - 最近导出记录
5. Axios 统一拦截器处理 token 失效和错误提示。

完成标准：
- 从登录到查询到导出都能在页面完成。
- 页面虽然不花哨，但流程完整且稳定。

预计工作量：10 到 12 小时。

---

## 第 9 步：联调 + 测试 + 修 Bug，别把问题留到最后

- 这一步做什么：系统化测试并修复关键问题
- 作用和意义：答辩失败往往不是功能没写，而是现场一跑就报错

详细操作：
1. 写一份最小测试清单：
   - 登录成功/失败
   - 手动采集
   - 定时采集
   - 查询分页
   - 导出文件
2. 后端写基础 Pytest（至少覆盖核心 service）。
3. 用 Swagger 或 Postman 跑所有核心接口。
4. 前端重点测：空数据、异常返回、token 过期。
5. 每修一个 Bug，记录到 docs/test.md。

完成标准：
- 主链路连续执行成功 3 次以上。
- 已知 Bug 有记录，有修复结论。

预计工作量：6 到 8 小时。

---

## 第 10 步：把课程要求文档补齐，避免“代码有了但没分”

- 这一步做什么：补全课程要求的所有文档
- 作用和意义：课程评分里文档占比高，缺文档会直接丢分

详细操作：
1. 逐个补齐这些文件：
   - user_stories.md
   - use_cases.md
   - architect.md
   - db.md
   - backend_api.md
   - ui_design.md
   - ai.md
   - assign.md
   - test.md
   - install.md
   - user_guid.md
2. ai.md 必须写清楚：提示词、AI 输出、你怎么改、为什么改。
3. assign.md 要记录每天做了什么。
4. install.md 必须让别人按步骤能启动成功。

完成标准：
- 文档完整可交。
- 文档内容和实际代码一致。

预计工作量：6 到 8 小时。

---

## 第 11 步：做 2 次完整演练

- 这一步做什么：固定演示顺序，压缩现场风险
- 作用和意义：演示稳定比“功能多一点”更重要

详细操作：
1. 固定演示脚本：
   - 登录
   - 手动触发采集
   - 看任务日志
   - 查股票/基金
   - 导出
   - 展示文档
2. 第一次演练：只看功能能不能跑通。
3. 第二次演练：计时，控制在 5 到 8 分钟。
4. 准备一个“异常兜底方案”：
   - 预置一批已采集数据
   - 网络波动时直接演示查询和导出

预计工作量：3 到 4 小时。
