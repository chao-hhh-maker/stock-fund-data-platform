## 2. 需求分析阶段材料

| 文件 | 必须内容 | 检查重点 |
| --- | --- | --- |
| `docs/user_stories.md` | 用户故事，格式为“作为一名...我想...以便...” | 覆盖管理员、普通用户、研究员/数据分析用户、运维人员 |
| `docs/use_cases.md` | 每个用户故事对应交互场景 | 至少覆盖登录、采集、查询、导出、监控、权限拦截、数据质量处理 |
| `docs/ai.md` | 需求分析中使用 AI 的提示词、输出摘要、问题、人工迭代 | 要保留“原始提示词”和“人工修改原因”，不能只写 AI 帮我生成 |

## 3. 设计阶段材料

| 文件 | 必须内容 | 检查重点 |
| --- | --- | --- |
| `docs/architect.md` | 架构设计、分层结构、核心类/模块职责 | 要体现前后端分离、服务层、任务调度、权限、安全、监控模块 |
| `docs/db.md` | ER 设计、表结构、字段说明、索引原因 | 必须与 `sql/init.sql` 和 ORM 模型一致，包含权限、审计、导出、告警表 |
| `docs/backend_api.md` | RESTful API 文档，最好对齐 OpenAPI/Swagger | 路径、方法、参数、响应、权限要求要和当前后端一致 |
| `docs/ui_design.md` | 前端页面设计、页面结构、交互说明 | 要同步最新升级后的页面：驾驶舱、行情、基金、公告、任务、数据源、质量、监控、元数据、SQL、导出、系统管理 |
| 数据库脚本说明 | `sql/init.sql` 的用途和执行方式 | 在文档中说明 SQLite 默认运行、MySQL 脚本用于课程交付/部署 |

## 4. AI 使用报告材料

| 事项 | 要求 | 建议准备 |
| --- | --- | --- |
| Prompt 示例 | 指导书要求附 Prompt 示例 | 选 6-10 个代表性 prompt：需求、用例、ER/API、前端页面、测试、Bug 修复、重构 |
| AI 输出摘要 | 记录 AI 给出的关键输出 | 不贴大段全文，摘要说明产出是什么 |
| 问题与人工修改 | 说明 AI 输出哪里不对、你如何判断并修改 | 这是评分重点，强调人在回路中，不盲信 AI |
| 截图材料 | 考核项提到 AI 生成物截图 | 准备聊天截图、代码 diff 截图、测试通过截图、页面截图 |
| AI 伦理说明 | 明确哪些段落/函数主要由 AI 生成或辅助 | 报告中说明人工审查、测试、重构过程；答辩要能解释代码逻辑 |

## 8. 答辩演示准备

| 事项 | 内容 | 建议 |
| --- | --- | --- |
| 演示脚本 | 固定 5-8 分钟流程 | 登录 -> 驾驶舱 -> 数据源 -> 采集任务 -> 股票/基金查询 -> 导出 -> 监控告警 -> 文档说明 |

## 9. 最后核对顺序

1. 项目能启动，`http://127.0.0.1:5173/` 和 `http://127.0.0.1:8000/docs` 可访问。
2. `python -m pytest` 通过，并截图保存。
3. `npm run build` 通过，并截图保存。
4. README、安装文档、使用说明与当前项目一致。
5. `ai.md` 有真实 Prompt、AI 输出摘要、人工修改和验证过程。
6. `assign.md` 有阶段/日期工作记录。
7. `test.md` 有测试用例、结果和 Bug 修复记录。
8. 课程设计报告 docx 已整合所有文档、截图、代码附录。
9. 答辩演示脚本已演练，数据库保留可展示数据。
10. 提交仓库前检查无敏感密钥、无无关临时文件。
