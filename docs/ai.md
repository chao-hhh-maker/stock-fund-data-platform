# AI 使用记录（ai.md）

> 贯穿模块 1–5 · 记录与 AI 有意义的交互：原始提示词、AI 输出摘要、存在的问题、迭代优化
> 课程要求：明确标注哪些代码 / 文档由 AI 生成，每位成员能解释自己负责的 AI 生成代码逻辑。

## 使用说明

本项目在「人在回路（Human-In-the-Loop）」原则下使用 AI：AI 负责生成初稿与样板代码，
团队成员负责审查、调试、重构与验证。下表按模块记录关键交互；标注 `[AI生成]` 的代码段
均经过人工审查与测试。

---

## 模块 1：需求分析

### 交互 1-1 理解题目
- **原始提示词**：「对于股票基金数据获取和管理平台，如何理解以下功能：多渠道数据源接入、数据清洗与标准化、数据查询与 API 服务、用户权限与安全管理？」
- **AI 输出摘要**：拆解出 6 大模块职责，指出对课程设计应聚焦「采集→清洗→存储→查询→导出→监控」最小闭环，进阶项（ClickHouse/多租户）可延后。
- **存在的问题**：AI 初版倾向企业级大而全，超出课程周期。
- **迭代优化**：补充「范围控制」约束，明确 MVP 与非 MVP 边界，落入 README 范围控制章节。

### 交互 1-2 生成用户故事
- **原始提示词**：「作为产品经理，为股票基金数据平台的『手动/定时采集』功能生成用户故事，遵循『作为一名…我想…以便…』格式。」
- **AI 输出摘要**：产出 US-03~US-07 采集相关用户故事及验收标准。
- **迭代优化**：人工补充 US-07「数据源不可用兜底」，这是真实演示风险点，AI 初版未覆盖。

## 模块 2：设计

### 交互 2-1 数据库 ER 设计
- **原始提示词**：「以题目二需求为输入，设计核心表并生成 ER 图与 MySQL 建表脚本，需包含用户、角色、标的、股票日线、基金净值、采集任务、运行记录、导出记录。」
- **AI 输出摘要**：生成 8 张表的 ER 图（Mermaid）与 init.sql。
- **存在的问题**：① `trigger` 字段名为 MySQL 保留字导致语法错误；② 缺少幂等所需的唯一约束。
- **迭代优化**：人工为 `trigger` 加反引号，补充 `(code, trade_date)` 唯一约束保证采集幂等。记为 BUG-02。

### 交互 2-2 API 设计
- **原始提示词**：「基于前后端分离，为上述需求生成 RESTful API 接口（OpenAPI 3.0 YAML），含认证、查询、采集、导出。」
- **AI 输出摘要**：生成接口清单与 OpenAPI 片段。
- **迭代优化**：人工统一分页结构 `{items,total,page,page_size}`，并明确每个接口的 RBAC 权限标注。

## 模块 3：编码实现

### 交互 3-1 采集器兜底逻辑（注释驱动生成）
- **原始提示词**：「// 采集股票日线：优先用 akshare 获取真实数据，失败时回退内置样例数据，单个代码失败不中断整体，记录数据来源 —— 请实现该函数。」
- **AI 输出摘要**：生成 `crawler._collect_one_stock` 等函数骨架。 `[AI生成]`（见 `app/services/crawler.py`）
- **存在的问题**：AI 初版把 akshare 导入放在模块顶层，无网络环境下启动即崩溃。
- **迭代优化**：人工改为函数内延迟导入，并用 try/except 包裹回退；补充 source 标记用于数据血缘。

### 交互 3-2 清洗与代码标准化
- **原始提示词**：「实现证券代码标准化函数，支持 sh600519 / 600519 / 600519.SH / 基金代码 等写法统一为『数字.市场后缀』。」
- **AI 输出摘要**：生成 `cleaning.normalize_code`。 `[AI生成]`
- **迭代优化**：人工补充基金 .OF 分支与边界用例，并据此编写单元测试反向验证。

### 交互 3-3 JWT 认证
- **原始提示词**：「// 验证用户名密码并返回 JWT token，token 含用户名与角色 —— 实现登录与鉴权依赖。」
- **AI 输出摘要**：生成 `security.create_access_token` 与 `deps.get_current_user`。 `[AI生成]`
- **存在的问题**：AI 默认用 bcrypt，Windows 上 bcrypt 原生库易安装失败。
- **迭代优化**：人工改用 passlib 的 `pbkdf2_sha256`（纯 Python），消除编译依赖。

### 交互 4-1 生成单元测试

### 交互 4-2 缺陷定位
- **原始提示词**：「启动报错 `error parsing value for field "CORS_ORIGINS" from source DotEnvSettingsSource`，分析原因并给出修复。」
- **AI 输出摘要**：指出 pydantic-settings 对 List 字段会强制 JSON 解析 env，逗号分隔字符串不被接受。
- **迭代优化**：采纳建议，改为字符串字段 + 解析属性。记为 BUG-01。

## 增强迭代（第二轮：对照题目逐模块做厚）

### 交互 E-1 缓存与限流的等价实现
- **原始提示词**：「题目要求 Redis 缓存和查询限流，但课设要保持一键运行不装 Redis。用进程内方案做架构等价实现，并预留可替换为 Redis 的接口。」
- **AI 输出摘要**：生成 `InMemoryTTLCache`（含命中率统计）与令牌桶 `RateLimiter`，抽象 `CacheBackend` 接口。
- **迭代优化**：人工补充按前缀失效 `invalidate_prefix`，用于数据更新后清查询缓存。

### 交互 E-2 数据完整性与节假日
- **原始提示词**：「实现数据完整性检查：检测每个标的近 30 个交易日的缺口，要排除周末和 A 股法定节假日。」
- **AI 输出摘要**：生成 `calendar_service`（节假日表 + 交易日计算）与 `monitor_service.check_integrity`。
- **存在的问题**：AI 初版把基金净值也按股票交易日要求，导致基金完整率虚低。
- **迭代优化**：人工对完整率取 `min(actual/expected, 1.0)` 上限，容忍基金披露滞后。

### 交互 E-3 akshare 新版签名缺陷
- **原始提示词**：「akshare 报错 `fund_open_fund_info_em() got an unexpected keyword argument 'fund'`，分析并修复。」
- **AI 输出摘要**：指出新版参数由 `fund` 改为 `symbol`，ETF 应使用 `fund_etf_fund_info_em`。
- **迭代优化**：人工按代码前缀区分 ETF（5 开头）与开放式基金，分别调用。记为 BUG-04。

### 交互 E-4 前端深色金融科技风
- **原始提示词**：「把前端改成深色金融科技风：深蓝底 + 霓虹青强调 + 毛玻璃卡片，仪表盘做成数据驾驶舱，股票加 K 线图。」
- **AI 输出摘要**：生成 `theme.css` 全局变量、`StatCard` 数字动画卡片、`KLineChart` 蜡烛+成交量、`MiniChart` 饼图/仪表盘。
- **迭代优化**：人工统一 A 股「涨红跌绿」配色，调整 ECharts 暗色 tooltip 与坐标轴样式。

### 交互 E-5 单元测试生成
- **原始提示词**：「为新增的缓存、限流、交易日历、行业分类、交叉验证各生成 Pytest 单元测试。」
- **AI 输出摘要**：生成 test_enhancements.py 与 test_monitor_api.py 共 21 个用例。
- **迭代优化**：人工补充令牌桶「不同用户独立」与缓存「按前缀失效」的边界用例。

## AI 使用反思

1. **AI 擅长**：生成样板代码、文档初稿、测试用例骨架、ER/API 设计、等价实现方案。
2. **AI 易错点**：依赖选择不考虑运行环境（bcrypt、顶层导入）、忽略保留字、库版本签名变化、倾向过度设计、业务规则细节（基金披露滞后）。
3. **人工不可替代**：环境适配、边界条件、演示稳定性兜底、业务正确性判断、实际运行验证。
4. **结论**：AI 显著提速，但每段生成代码都经人工审查、调试与测试验证（38 个测试全过），未盲目采纳。
